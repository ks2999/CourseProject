## 1. Основные планы и этапы проекта

### 1.1. Краткое описание проекта

- **Название проекта:**  
  Платформа для обучения программированию с элементами геймификации

- **Цель проекта:**  
  Разработка веб‑приложения для обучения программированию на языке C с использованием элементов геймификации, позволяющего студентам проходить теоретические материалы, решать задачи и отслеживать свой прогресс, а преподавателям — управлять учебным контентом и контролировать успеваемость.

- **Краткое описание задач:**  
  - Анализ аналогичных образовательных платформ и систем с геймификацией.  
  - Обоснование и выбор технологического стека для серверной и клиентской частей приложения.  
  - Проектирование архитектуры системы на основе микросервисного подхода.  
  - Разработка серверной части: моделирование предметной области, реализация REST API, аутентификации и авторизации, логики проверки решений и геймификации.  
  - Разработка клиентской части: веб‑интерфейс для студентов и преподавателей, интеграция с backend‑сервисами.  
  - Реализация механизмов геймификации (опыт, уровни, достижения, челленджи).  
  - Подготовка и загрузка теоретических материалов и набора практических задач.  
  - Тестирование, отладка и подготовка проектной документации.

## 2. Используемый технологический стек и его обоснование

### 2.1. Перечень используемых технологий

| Технология / Инструмент | Описание | Причины выбора |
|-------------------------|----------|----------------|
| Java 17 | Основной язык реализации серверной части | Современная LTS‑версия, широкая поддержка, совместимость со Spring Boot |
| Spring Boot 3 (Web, Security, Data JPA) | Фреймворк для разработки веб‑приложений и микросервисов | Упрощённая конфигурация, богатая экосистема, встроенная поддержка REST, безопасности и работы с БД |
| Spring Security + JWT | Механизм аутентификации и авторизации пользователей | Стандарт де‑факто в Java‑экосистеме, гибкая настройка ролей и прав, удобная интеграция с REST‑API |
| Spring Data JPA / Hibernate | Уровень доступа к данным и ORM | Ускоряет разработку CRUD‑операций, снижает объём шаблонного SQL‑кода, хорошо интегрируется со Spring Boot |
| PostgreSQL | Реляционная СУБД | Надёжная, производительная, свободно распространяемая СУБД, удобна для учебных и промышленных проектов |
| Flyway | Инструмент миграций базы данных | Контролируемые изменения схемы БД, воспроизводимость структуры данных в разных окружениях |
| HTML, CSS, JavaScript (Fetch API) | Технологии клиентской части веб‑приложения | Стандартный стек для браузерных приложений, простая интеграция с REST‑API |
| (Опционально) Docker, Docker Compose | Контейнеризация сервисов и инфраструктуры | Удобство локального развёртывания и демонстрации, изоляция окружения; средство, а не цель проекта |

---

### 2.2. Обоснование выбранного технологического стека

Технологический стек ориентирован на разработку промышленно‑ориентированного серверного приложения на Java с использованием Spring Boot как стандартного фреймворка для веб‑сервисов. Это позволяет реализовать микросервисную архитектуру, REST‑API, аутентификацию и авторизацию на основе JWT, а также удобную работу с реляционной базой данных через Spring Data JPA и Hibernate.

PostgreSQL выбрана как надёжная и функциональная СУБД с открытой лицензией, подходящая для хранения учебных материалов, задач, результатов тестов и данных геймификации. Flyway обеспечивает контролируемые миграции схемы БД и воспроизводимость структуры данных в разных окружениях.

Использование HTML/CSS/JavaScript на стороне клиента позволяет реализовать удобный веб‑интерфейс, в котором студент может писать код прямо в браузере, отправлять решения на сервер и сразу видеть результаты проверки.  
Docker и Docker Compose при необходимости упрощают развёртывание всего комплекса сервисов и инфраструктуры, но не являются самостоятельной целью проекта.

В сумме выбранный стек:
- обеспечивает реализацию всех ключевых функциональных требований (уроки, задачи, проверка решений, геймификация);
- поддерживает требования по безопасности (управление доступом, роли, токены);
- позволяет развивать и масштабировать систему в дальнейшем.

## 3. Критерии оценивания проекта

Ниже приведён набор количественных критериев, по которым может оцениваться прогресс и качество проекта.

| Критерий | Описание |
|----------|----------|
| Функциональность — процент выполнения требований | Доля реализованных и корректно работающих функциональных требований от общего количества, указанных в ТЗ (в процентах) |
| Работа с базой данных | Наличие и корректность операций чтения/записи/обновления/удаления для основных сущностей (пользователи, уроки, задачи, решения, прогресс, достижения) |
| Управление доступом | Корректность разграничения прав доступа между ролями (студент, преподаватель, администратор), защита API с помощью Spring Security и JWT |
| Качество проверки решений задач | Доля задач, для которых реализованы автоматические тесты, корректно определяющие правильные и неправильные решения |
| Использование элементов геймификации | Количество реализованных механизмов геймификации (XP, уровни, достижения, челленджи) от общего числа запланированных |
| Документация и оформление | Наличие README с описанием проекта, архитектуры и структуры репозитория, а также отдельного документа по типу «Текста программы» |

При необходимости из этого перечня для формы КТ можно выбрать 3–5 наиболее релевантных критерия.

## 4. Особые пометки

На момент КТ1 основное внимание уделяется:
- формированию требований и пользовательских сценариев;
- проектированию архитектуры и структуры данных;
- реализации базового каркаса backend‑сервиса и подготовке структуры репозитория.

К моменту завершения проекта планируется:
- реализовать полный цикл работы студента (уроки, задачи, отправка решений, получение обратной связи, прогресс, достижения);
- обеспечить базовые инструменты для преподавателя (создание уроков, задач, соревнований, просмотр прогресса);
- внедрить ключевые элементы геймификации (опыт, уровни, достижения, челленджи).

Отдельные элементы, такие как streak (серия дней), полноценный лидерборд по группе и автоматическая «задача дня», рассматриваются как направления для дальнейшего развития и могут быть частично реализованы или описаны в разделе перспектив развития проекта.
